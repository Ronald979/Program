# 模型選型決策表（面試口袋版）

> 目標：在 RTX 1050 的條件下，面試時可以「快速做出合理模型選擇」並解釋 trade-off。

## 快速判斷邏輯
1. 先判斷任務：生成文字 / 檢索 / 分類 / CV / 語音 / 排序  
2. 再判斷資源：可用 VRAM、是否可接受延遲、是否可接受精度犧牲  
3. 最後判斷維運：是否需要可微調、是否要多人共用、是否有法規限制  

## 建議對照表

| 任務場景 | 首選模型族群 | 推薦規模（本地） | 為什麼選它 | 成本/延遲/精度 | RTX 1050 實務 |
|---|---|---|---|---|---|
| 對外客服/提問回覆（中文/英文） | LLM 小模型 + RAG | 1B~7B（量化） | 能直接回應、可透過 prompt 控制 | 成本低、延遲高但可控、精度中上 | 用 1-2B 量化版作主力，遇到難題降級或固定模板 |
| 文件 QA（內部知識庫） | Embedding + cross-encoder 重排序 | embedding 小模型 + 小型 reranker | RAG 成果穩定，追求可追溯 | 向量檢索快、可解釋、精度高於關鍵字 | `sentence-transformers` 小模型可行，重排序可以 batch 小量離線 |
| 文本分類 / 意圖辨識 | DistilBERT / MiniLM / Logistic | 50M~200M | 訓練快、延遲可控 | 低成本、精度中上、可解釋 | 較適合作為基準線模型 |
| CV 缺陷辨識 / 小型物件 | YOLOv8n / MobileViT / ResNet 18 | 小模型 | 上線穩定、推論快、標註資料可導向 | 成本較低、精度取決於資料品質 | 若無 CUDA 能力可先做 CPU + 小 batch，保留可重用資料流 |
| 語音轉文字 | Whisper tiny/base / Vosk | tiny/base | 可直接服務化，容易上線 | 速度快於大模型，精度中高 | tiny 可跑較快，小型應用較實用 |
| 推薦排序（內容/列表） | LightFM / CatBoost / logistic 回歸 | 小到中 | 易上線、可解釋、可與規則融合 | 成本低、維護容易、精度可漸進 | 適合做 baseline，搭配 AB 測試 |
| 搜尋 / 召回前處理 | TF-IDF + 向量混和 | - | 成本最省，覆蓋冷啟動 | 非常低成本，精度較一般 | 先做傳統特徵後接向量，避免一開始過度複雜 |
| 高風險高精度專案（面試展示） | 適度大小模型 + 流程化治理 | 視硬體 | 重點在「完整流程」而非模型神蹟 | 成本高，需更完整治理 | 以 1050 只能做示範與比較，不宜追求壓倒性效能 |

## 2 分鐘面試口頭版（背下就能說）
- 當你問：為什麼不用最大模型？  
  - 答：優先看任務、延遲、成本與穩定性；未必大模型最好，基準化比較後才升級。  
- 當你問：為什麼要用 embedding？  
  - 答：因為可追溯、可控、可減少幻覺，特別適合知識庫導向需求。  
- 當你問：如何在 1050 上跑？  
  - 答：先量化、減小 batch、降低上下文長度、做 fallback 策略，必要時採模型路由。  

## 模型組合建議（面試最受點）
- `LLM 主線 + Embedding + API Gateway`：能解釋「多模型路由」與降級。
- `LLM 主線 + SQL + 向量 index + cache`：能解釋「低延遲保護」能力。
- `LLM + 小模型分類`：能解釋「把大模型保留在關鍵路徑」的架構思路。
